# Entity Types

## What is an Entity?

An entity is any managed unit in the layer-stage system: projects, features, components, stages, sub-layers, etc. All entities follow the canonical structure defined in `.0agnostic/06_context_avenue_web/01_file_based/04_@import_references/entity_structure.md`.

## Entity Types

### 1. Projects (layer_1)

**Purpose**: A complete application, system, or body of work

**Structure** (see `entity_structure.md` for full canonical tree):
```
layer_1_project_<name>/
├── 0AGNOSTIC.md              # Identity (source of truth)
├── 0INDEX.md                 # Contents index
├── README.md                 # Human-readable overview
├── CLAUDE.md                 # Generated by agnostic-sync.sh
├── AGENTS.md, GEMINI.md, OPENAI.md, .cursorrules  # Generated
├── .0agnostic/               # AI resources (numbered subdirectories)
│   ├── 01_knowledge/         # Domain knowledge (per-topic dirs)
│   ├── 02_rules/{static,dynamic}/
│   ├── 03_protocols/
│   ├── 04_episodic_memory/{sessions,changes}/
│   ├── 05_handoff_documents/{01_incoming,02_outgoing}/
│   ├── 06_context_avenue_web/{01_file_based,02_data_based}/
│   └── 07+_setup_dependant/
├── .1merge/                  # Tool overrides (6 tools x 3 tiers)
├── .claude/{rules,episodic_memory/}
├── .cursor/{rules,episodic_memory/}
├── .gemini/, .codex/, .github/
├── layer_1_group/            # Internals
│   ├── layer_1_00_layer_registry/proposals/
│   └── layer_1_99_stages/    # All stages (00-11)
├── layer_2_group/            # Children (features)
│   ├── layer_2_00_layer_registry/proposals/
│   └── layer_2_features/
```

### 2. Features (layer_2+)

**Purpose**: A distinct capability within a project

**Structure**:
```
layer_2_feature_<name>/
├── 0AGNOSTIC.md, 0INDEX.md, README.md
├── CLAUDE.md, AGENTS.md, GEMINI.md, OPENAI.md  # Generated
├── .0agnostic/               # Full numbered structure (01-07+)
├── .1merge/                  # 6 tools x 3 tiers
├── .claude/, .cursor/, .gemini/, .codex/, .github/
├── layer_2_group/            # Feature internals
│   ├── layer_2_00_layer_registry/proposals/
│   └── layer_2_99_stages/    # All stages (00-11)
├── layer_3_group/            # Children (components)
│   ├── layer_3_00_layer_registry/proposals/
│   └── layer_3_components/
```

### 3. Research Projects (layer_-1)

**Purpose**: Experimental, exploratory work

**Structure**:
```
layer_-1_<name>/
├── 0AGNOSTIC.md, 0INDEX.md, README.md
├── CLAUDE.md, AGENTS.md, GEMINI.md, OPENAI.md  # Generated
├── .0agnostic/               # Full numbered structure (01-07+)
├── .1merge/                  # 6 tools x 3 tiers
├── .claude/, .cursor/, .gemini/, .codex/, .github/
├── layer_-1_group/           # Research internals
│   ├── layer_-1_00_layer_registry/proposals/
│   └── layer_-1_99_stages/   # All stages (00-11)
├── layer_0_group/            # Research features
│   ├── layer_0_00_layer_registry/proposals/
│   └── layer_0_features/
```

### 4. Setup-Dependant Entities

**Purpose**: Machine/OS/environment-specific configuration within `07+_setup_dependant/`

**Structure**: Same as any other entity, EXCEPT:
- **No `07+_setup_dependant/` inside their `.0agnostic/`** — they're already within one
- Knowledge topics replace what would have been old-style sub-layers (e.g., `ubuntu_desktop/`, `system_services/` as dirs under `.0agnostic/01_knowledge/`)
- Internal stages track setup work (audio, display, services, etc.)
- May have child entities for further specificity (e.g., Local → Coding Apps → Cursor)

```
sub_layer_0_06_local/              # Example: local Ubuntu environment
├── 0AGNOSTIC.md, 0INDEX.md, README.md
├── CLAUDE.md (etc.)               # Generated
├── .0agnostic/                    # Numbered dirs, NO 07+_setup_dependant
│   ├── 01_knowledge/              # Knowledge TOPICS (not old-style sub-layers)
│   │   ├── linux_fundamentals/    # Topic with docs/, principles/, resources/
│   │   ├── ubuntu_desktop/
│   │   ├── system_services/
│   │   └── audio/
│   ├── 02_rules/{static,dynamic}/
│   ├── 03_protocols/
│   ├── 04_episodic_memory/
│   ├── 05_handoff_documents/
│   └── 06_context_avenue_web/
├── .1merge/, .claude/, .cursor/ (etc.)
├── sub_layer_0_06_group/          # Internal group
│   ├── sub_layer_0_06_00_layer_registry/proposals/
│   ├── sub_layer_0_06_99_stages/  # Setup stages (00-11)
│   └── setup/                     # Setup-specific content (instruction docs, etc.)
└── sub_layer_0_07_group/          # Children group
    ├── sub_layer_0_07_00_layer_registry/proposals/
    └── sub_layer_0_07_coding_apps/  # Child entity
```

### 5. Stages

**Purpose**: Workflow phase container

**Structure** (see `entity_structure.md` for full canonical tree):
```
stage_N_XX_<name>/
├── 0AGNOSTIC.md              # Stage-level context
├── CLAUDE.md (etc.)          # Generated by agnostic-sync.sh
├── .0agnostic/               # Full numbered structure (same as entity root)
├── .1merge/                  # 6 tools x 3 tiers
├── .claude/, .cursor/ (etc.)
├── outputs/                  # Stage work products
└── synthesis/                # Stage synthesis
```

### 6. Proposals

**Purpose**: Formal change proposals with staging

**Structure**:
```
proposals/
├── 0INDEX.md
├── active/                   # Production proposals
├── staging/                  # Under testing
│   ├── stage_experimental/
│   ├── stage_testing/
│   └── stage_rollout/
└── archived/                 # Historical
```

## Deprecated Patterns

The following patterns are **no longer used**. If found in existing entities, they should be migrated:

| Old Pattern | Replacement |
|-------------|-------------|
| `layer_N_01_ai_manager_system/` | Entity root `0AGNOSTIC.md` (identity lives there now) |
| `layer_N_02_manager_handoff_documents/` | `.0agnostic/05_handoff_documents/` |
| `layer_N_03_sub_layers/` | `.0agnostic/01_knowledge/` (knowledge topics) + `.0agnostic/02_rules/` + `.0agnostic/03_protocols/` |
| `sub_layer_N_XX_knowledge_system/` | `.0agnostic/01_knowledge/` |
| `sub_layer_N_XX_rules/` | `.0agnostic/02_rules/` |
| `sub_layer_N_XX_protocols/` | `.0agnostic/03_protocols/` |
| `sub_layer_N_XX_prompts/` | `.0agnostic/06_context_avenue_web/01_file_based/05_skills/` |
| Topic sub-layer directories | `.0agnostic/01_knowledge/[topic]/` (with `principles/`, `docs/`, `resources/`) |

**Migration tool**: `.0agnostic/01_knowledge/layer_stage_system/resources/tools/migrate-sub-layers-to-0agnostic.sh`

## Entity File Requirements

### Required Files

| File | Purpose | Required For |
|------|---------|--------------|
| `0AGNOSTIC.md` | Identity, triggers, pointers (source of truth) | All entities |
| `0INDEX.md` | Contents listing / navigation | All entities |
| `README.md` | Human-readable overview | All entities |
| `CLAUDE.md` etc. | Tool-specific context (auto-generated) | All entities |

### Key Conventions

- **Internal group** always uses `_group` suffix: `layer_N_group/`
- **Children group** also uses `_group` suffix: `layer_N+1_group/`
- **Entity resources** live in `.0agnostic/` with numbered subdirectories (01-07+)
- **Knowledge** organized per-topic: `01_knowledge/[topic]/{principles,docs,resources}`
- **Episodic memory** always named `episodic_memory/` (NOT `episodic/`)
- **Handoff documents** in `.0agnostic/05_handoff_documents/` (NOT in `layer_N_group/`)
- **Outputs** and **synthesis** live inside each **stage**, NOT at entity root
- Setup-dependant entities omit `07+_setup_dependant/` from their `.0agnostic/`

## Entity Nesting Rules

1. **Projects contain features**: `layer_1` → `layer_2`
2. **Features contain components**: `layer_2` → `layer_3`
3. **Research uses layer_0 for features**: `layer_-1` → `layer_0`
4. **Setup-dependant entities nest by specificity**: OS → distro → environment → local → apps
5. **Stages don't nest**
