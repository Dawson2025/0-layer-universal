# Cursor Context Mapping — Layer-Stage System Integration

*How the agnostic system delivers context to Cursor IDE*

## Current State

### What agnostic-sync.sh Generates for Cursor

| File | Content | Format |
|------|---------|--------|
| `.cursorrules` | Lean: Identity + Navigation sections only | Plain markdown |

The `.cursorrules` file is generated by `agnostic-sync.sh` using the `generate_lean()` function, extracting only `Identity` and `Navigation` sections from `0AGNOSTIC.md`.

### How Cursor Loads It

1. Cursor reads `.cursorrules` from the repo root at session start
2. Content is included in the AI model's context for every interaction
3. **Warning**: `.cursorrules` is deprecated — Cursor Agent mode may ignore it

## Migration Plan: .cursorrules → .cursor/rules/*.mdc

### Phase 1: Parallel Generation (Recommended Next Step)

Update `agnostic-sync.sh` to generate BOTH `.cursorrules` (backward compat) and `.cursor/rules/` (modern):

```bash
# In agnostic-sync.sh, add after .cursorrules generation:
mkdir -p "$DIR/.cursor/rules"

# Identity rule — always applied
cat > "$DIR/.cursor/rules/identity.mdc" << MDC
---
description: Entity identity and role for this directory
alwaysApply: true
---
$(extract_named_sections "$STATIC_CONTENT" "Identity")
MDC

# Triggers rule — applied intelligently
cat > "$DIR/.cursor/rules/triggers.mdc" << MDC
---
description: When to load specific skills or resources
alwaysApply: false
---
$(extract_named_sections "$STATIC_CONTENT" "Triggers")
MDC

# Navigation rule — applied intelligently
cat > "$DIR/.cursor/rules/navigation.mdc" << MDC
---
description: How to navigate the layer-stage hierarchy
alwaysApply: false
---
$(extract_named_sections "$STATIC_CONTENT" "Navigation")
MDC
```

### Phase 2: Hot Rule Promotion to .mdc

Each promoted rule gets its own `.mdc` file:

```
.cursor/rules/
  promoted/
    file_change_reporting.mdc    # alwaysApply: true
    ai_context_modification.mdc  # alwaysApply: true
```

### Phase 3: Dynamic Rules as Intelligent Rules

Dynamic rules map naturally to Cursor's "Apply Intelligently" type:

```
.cursor/rules/
  dynamic/
    browser_extraction.mdc       # description: "When working with browser automation"
    source_of_truth.mdc          # description: "When editing context files"
    parallel_stages.mdc          # description: "When multiple stages need concurrent work"
```

### Phase 4: Deprecate .cursorrules

Once `.cursor/rules/` generation is stable:
1. Stop generating `.cursorrules` in `agnostic-sync.sh`
2. Add `.cursorrules` to `.gitignore`
3. Update documentation

## Rule Priority Mapping

| Layer-Stage Concept | Cursor Rule Type | Applied |
|---------------------|-----------------|---------|
| `0AGNOSTIC.md` Identity | Project Rule (alwaysApply) | Every session |
| Hot promoted rules | Project Rule (alwaysApply) | Every session |
| Static rules | Project Rule (alwaysApply) | Every session |
| Dynamic rules | Project Rule (intelligently) | When relevant |
| Scenario-based rules | Project Rule (intelligently) | When relevant |
| Stage-specific context | Project Rule (file globs) | When in stage dirs |

## AGENTS.md Compatibility

Cursor also reads `AGENTS.md` from repo root. Since `agnostic-sync.sh` already generates `AGENTS.md`, Cursor gets context from both:
1. `.cursorrules` (or `.cursor/rules/`) — structured rules
2. `AGENTS.md` — full static content (same as CLAUDE.md but with AutoGen boilerplate)
