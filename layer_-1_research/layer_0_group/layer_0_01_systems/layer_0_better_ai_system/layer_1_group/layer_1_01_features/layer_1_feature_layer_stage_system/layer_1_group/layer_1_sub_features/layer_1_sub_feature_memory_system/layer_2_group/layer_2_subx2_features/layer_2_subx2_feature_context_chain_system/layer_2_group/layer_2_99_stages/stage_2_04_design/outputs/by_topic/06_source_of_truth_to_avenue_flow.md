## Numbering Encodes the Flow

| Number | Name | Role in Flow |
|--------|------|--------------|
| 01 | knowledge/ | Content: what we know |
| 02 | rules/ | Content: what we must do |
| 03 | protocols/ | Content: how we do things |
| 04 | episodic_memory/ | Content: what happened |
| 05 | handoff_documents/ | Content: what to pass on |
| 06 | context_avenue_web/ | **Delivery**: how content reaches consumers |
| 07+ | setup_dependant/ | **Environment**: OS/tool configuration |

**01-05 are producers.** They create and store content. They are edited by humans and agents.

**06 is the delivery mechanism.** It transforms and indexes content for consumption. It is generated by sync scripts.

**07+ is the environment layer.** It configures how the system runs on specific platforms.

## Context Loading Order (Agent Perspective)

When an AI agent enters an entity, it loads context in this order:

1. **CLAUDE.md / tool file** — auto-loaded by tool runtime, gives identity + critical rules
2. **Path rules** — auto-loaded on directory entry, gives scoped behavioral constraints
3. **Skills check** — match WHEN conditions against current task
4. **0AGNOSTIC.md STATIC** — if agent needs more than tool file provides
5. **0AGNOSTIC.md DYNAMIC** — on-demand when agent needs state details
6. **.0agnostic/ resources** — read specific knowledge, protocols, episodic as needed
7. **Data-based avenues** — query for semantic search, dashboards, temporal context (if available)

Steps 1-6 work with zero dependencies (file-based only). Step 7 is an optional accelerator.

---

## Key Architectural Properties

**Single direction of authority**: Source of truth (01-05) → Avenues (06). Never the reverse.

**Graceful degradation**: If databases are down, file-based avenues still work. Zero-dependency operation is always preserved.

**Consistent ordering convention**: Both file-based and data-based avenues follow the same comprehensiveness ordering, making the system predictable and navigable.
